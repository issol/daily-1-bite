Claude Code를 어느 정도 써온 분들이라면 이 벽에 부딪혀봤을 겁니다. Claude가 특정 작업을 잘하는 건 알겠는데, 매번 터미널을 열고 직접 실행해줘야 한다는 문제입니다. 매일 밤 문서 감사, 매일 아침 PR 트리아지, 매주 의존성 점검 — 모두 유용한 작업이지만, 모두 사람이 시작해줘야 합니다.

Anthropic이 2026년 4월 14일에 **Claude Code Routines** 리서치 프리뷰를 공개했습니다. 이 기능은 그 문제의 답입니다. Routines는 Claude가 스케줄, API 호출, GitHub 이벤트를 트리거로 — Anthropic이 관리하는 클라우드 인프라에서 — 자율적으로 실행되도록 합니다. 내 노트북이 켜져 있지 않아도 됩니다.

## Routine이 정확히 뭔가요?

Routine은 4가지 구성요소로 이루어진 저장된 설정입니다.

1. **프롬프트** — Claude에게 시킬 작업
2. **레포지토리** — Claude가 접근할 저장소
3. **커넥터** — 연동할 외부 서비스 (Slack, Linear, Google Drive, GitHub)
4. **트리거** — Routine을 실행시키는 조건

한 번 설정하면 Claude가 자율적으로 실행합니다. 실행 중 수동 승인이 필요 없고, Anthropic 인프라에서 돌아갑니다. 내 노트북이 켜져 있을 필요가 없습니다.

## 트리거 3가지

### 1. 스케줄 (Schedule)

가장 단순한 트리거입니다. 주기를 설정(매시간, 매일 밤, 매주)하면 Claude가 해당 주기에 프롬프트를 실행합니다.

```
트리거: 스케줄
주기: 매일 밤 (UTC 02:00)
프롬프트: "지난 24시간 동안 새로 올라온 GitHub 이슈를 전부 확인해줘.
           유형별(버그/기능 요청/질문)로 분류한 뒤
           Slack #triage 채널에 요약해서 올려줘."
```

대부분의 팀이 이런 작업을 위해 크론 잡 + 쉘 스크립트 + Slack 웹훅을 조합해 쓰는데, 그 스택을 대체할 수 있습니다.

### 2. API

Routine마다 고유한 HTTP 엔드포인트가 생성됩니다. CI 파이프라인, 내부 툴링, 어디서든 POST를 날리면 Routine이 실행됩니다.

```bash
# API로 Routine 트리거
curl -X POST https://api.claude.ai/code/routines/{routine-id}/trigger \
  -H "Authorization: Bearer $CLAUDE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"context": "프로덕션 배포 완료"}'
```

기존 워크플로우의 이벤트(배포 완료, 테스트 통과, 티켓 상태 변경 등)에 Claude의 판단을 연결하고 싶을 때 유용합니다. Claude Code 안에서 모든 걸 새로 만들 필요 없이, 기존 파이프라인에서 호출하기만 하면 됩니다.

### 3. GitHub 웹훅

개발자 입장에서 가장 흥미로운 트리거입니다. Routine을 GitHub 저장소와 연결하면 저장소 이벤트에 반응해 자동 실행됩니다. 지원 이벤트: `pull_request`, `push`, `issue`, `check_run`, `workflow_run`, `discussion`, `release`, `merge_queue`.

```
트리거: GitHub 웹훅
이벤트: pull_request.opened
프롬프트: "이 PR을 보안 이슈, API 계약 변경, 테스트 누락 여부 기준으로 검토해줘.
           결과를 심각도별로 정리해서 PR에 코멘트로 달아줘."
```

이게 Routines가 실제로 강력한 이유입니다. 정적 린트 검사 대신 Claude의 실질적인 판단이 PR마다 자동으로 달립니다. 리뷰어가 기억해서 직접 실행할 필요가 없습니다.

## 커넥터: Claude가 읽고 쓸 수 있는 것들

레포지토리 외에 연동 가능한 커넥터:

- **Slack** — 메시지 전송, 채널 히스토리 읽기
- **Linear** — 이슈 읽기 및 업데이트
- **Google Drive** — 문서, 스프레드시트 읽기
- **GitHub** — 트리거 저장소 외 추가 저장소 접근, 코멘트 작성, 이슈 생성, PR 업데이트

팀 자동화 대부분은 Slack + GitHub 조합으로 커버됩니다.

## 플랜별 사용 한도

| 플랜 | 일일 실행 횟수 |
|------|--------------|
| Pro | 5회 |
| Max | 15회 |
| Team / Enterprise | 25회 |

Routines 사용에는 웹에서 Claude Code가 활성화되어 있어야 합니다. `claude.ai/code/routines`에서 관리하거나 CLI에서 `/schedule` 명령으로 설정할 수 있습니다.

## 실무 활용 예시

**매일 밤 백로그 트리아지**
```
트리거: 스케줄 (매일 밤)
프롬프트: "레포의 모든 오픈 이슈를 확인해줘.
           14일 이상 활동이 없는 이슈에는 'needs-triage' 라벨을 달고,
           Slack #engineering-triage에 요약을 올려줘."
```

**PR 보안 리뷰**
```
트리거: GitHub 웹훅 (pull_request.opened)
프롬프트: "이 PR을 OWASP Top 10 취약점, 하드코딩된 시크릿,
           안전하지 않은 의존성 기준으로 검토해줘.
           결과를 심각도 등급과 함께 PR 리뷰 코멘트로 달아줘."
```

**배포 후 검증**
```
트리거: API (CI에서 배포 완료 후 호출)
프롬프트: "프로덕션 헬스 엔드포인트를 확인하고, 최신 배포 로그에서
           에러 패턴을 스캔해줘. 결과를 #deployments에 올리고,
           즉시 확인이 필요한 사항은 별도로 표시해줘."
```

**주간 문서 드리프트 감지**
```
트리거: 스케줄 (매주 월요일 09:00)
프롬프트: "/docs의 API 문서와 /src/routes의 실제 라우트를 비교해줘.
           문서에는 있지만 삭제된 엔드포인트, 구현됐지만 문서에 없는
           엔드포인트를 목록으로 만들어 GitHub 이슈로 등록해줘."
```

## GitHub Actions이나 크론과 뭐가 다른가요?

솔직하게 말하면, Routines가 GitHub Actions를 대체하지는 않습니다. "테스트 실행 → 커버리지 확인 → 기준 미달 시 실패" 같은 결정론적 자동화라면 GitHub Actions가 낫습니다. 더 빠르고, 감사 추적도 명확하고, 이런 종류의 작업에 잘 맞습니다.

Routines가 유리한 상황은 작업에 **판단**이 필요할 때입니다. "이 이슈들을 트리아지해줘"는 YAML 워크플로우로 표현하기 어렵습니다. "이 PR의 보안 이슈를 검토해줘"는 bash 스크립트로 만들 수 없습니다. Claude의 이해력이 실제로 필요한 작업, 그게 Routines가 GitHub Actions가 줄 수 없는 걸 제공하는 영역입니다.

실용적인 기준은 이렇습니다: 원래 사람이 해야 할 작업인가요? 그렇다면 Routines가 적합할 수 있습니다. 순수 자동화 로직이라면 Actions를 유지하세요.

## 리서치 프리뷰의 제약 사항

리서치 프리뷰 단계이므로 몇 가지 주의할 점이 있습니다.

- 일일 실행 횟수 제한(5~25회)은 변경될 수 있음
- 일부 커넥터는 지역에 따라 아직 미지원
- GitHub 웹훅 이벤트 목록은 확장될 예정 (현재 지원: `pull_request`, `push`, `issue`, `check_run`, `workflow_run`, `discussion`, `release`, `merge_queue`)
- Routine 실행 감사 로그는 웹 UI에서 확인 가능하지만 API를 통한 내보내기는 아직 미지원

## 5분 안에 설정하기

1. `claude.ai/code/routines`로 이동
2. **New Routine** 클릭
3. 프롬프트 작성 (결과 형식을 구체적으로 지정할수록 좋음)
4. 레포지토리 추가
5. 커넥터 추가 (Slack, Linear 등 필요한 서비스 권한 부여)
6. 트리거 설정
7. 저장 및 활성화

또는 CLI로:

```bash
# Routine 설정 마법사 시작
claude /schedule
```

CLI 마법사가 단계별로 안내해줍니다.

Routines에서 가장 흥미로운 건 스케줄링 기능 자체가 아닙니다. Anthropic이 Claude Code를 어떤 방향으로 가져가려는지를 보여준다는 점입니다. "직접 실행하는 터미널 툴"에서 "한 번 설정하는 클라우드 자동화 플랫폼"으로의 전환은 상당히 큰 방향 전환입니다. 일일 5~25회 실행 횟수가 팀 워크플로우에 충분한지는 사용 사례에 따라 다르겠지만, 기반 자체는 탄탄합니다.
